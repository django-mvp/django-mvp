# Quickstart: BaseTemplateNameMixin & PageMixin

**Branch**: `003-base-mixin-classes` | **Date**: 2026-05-02

This guide covers the two foundational view mixins in `mvp.views.base` that underpin all `django-mvp` view classes.

---

## BaseTemplateNameMixin

### What it does

Extends Django's template resolution to support a **mandatory fallback base template**. When `get_template_names()` runs, it returns the view-specific template(s) first (higher specificity), then appends `base_template_name` as a catch-all shell template.

This allows view classes to work even without a custom model-specific template — they fall back to a generic base layout.

### Usage

```python
from django.views.generic import DetailView
from mvp.views.base import BaseTemplateNameMixin

class MyDetailView(BaseTemplateNameMixin, DetailView):
    model = MyModel
    base_template_name = "my_detail_base.html"  # required — must be set
    # template_name = "myapp/mymodel_detail.html"  # optional — more specific
```

Django tries templates in this order:
1. `myapp/mymodel_detail.html` (auto-generated by Django, most specific)
2. `my_detail_base.html` (fallback from `base_template_name`)

### Rules

- `base_template_name` **must** be set on every concrete subclass.  
  Leaving it as `None` (the default) raises `ImproperlyConfigured` at render time.
- The base template is always appended **last** — view-specific templates always take priority.

### Override `get_template_names()`

To customise template resolution further:

```python
class ConditionalDetailView(BaseTemplateNameMixin, DetailView):
    model = MyModel
    base_template_name = "my_detail_base.html"

    def get_template_names(self):
        names = super().get_template_names()  # includes base_template_name
        if self.object.is_featured:
            names.insert(0, "myapp/featured_detail.html")
        return names
```

## PageMixin

### What it does

Injects a `page` context dict into every template. The dict groups all page-level rendering metadata under a single key, keeping the main context namespace clean and making it easy to identify where each context variable comes from.

```python
# In your template:
{{ page.title }}
{{ page.subtitle }}
{{ page.icon }}
{{ page.class }}
{% for crumb in page.breadcrumbs %}...{% endfor %}
```

### Declarative usage (class attributes)

```python
from mvp.views.base import PageMixin
from django.views.generic import TemplateView

class AboutView(PageMixin, TemplateView):
    template_name = "about.html"
    page_title = "About Us"
    page_subtitle = "Learn more about our team"
    page_icon = "fas fa-info-circle"
    page_class = "about-page"
    breadcrumbs = [
        {"text": "Home", "href": "/"},
        {"text": "About"},           # no href = current page
    ]
```

### Dynamic usage (method overrides)

Override any `get_*()` method when the value depends on request state, the object being viewed, or other runtime context:

```python
from django.utils.translation import gettext_lazy as _

class ProductDetailView(PageMixin, BaseTemplateNameMixin, DetailView):
    model = Product
    base_template_name = "detail_view.html"

    def get_page_title(self):
        return self.object.name  # dynamic: uses the resolved object

    def get_breadcrumbs(self):
        return [
            {"text": _("Home"), "href": "/"},
            {"text": _("Products"), "href": "/products/"},
            {"text": self.object.name},
        ]
```

### Attribute vs. method: when to use which

| Situation | Recommended approach |
|-----------|---------------------|
| Static string known at class definition time | Class attribute (`page_title = "Dashboard"`) |
| Value depends on the current object (`self.object`) | Override `get_page_title()` |
| Value depends on request data or user | Override `get_page_title()` |
| Breadcrumbs are always the same | `breadcrumbs = [...]` class attribute |
| Breadcrumbs include the current object's name or URL | Override `get_breadcrumbs()` |

### `page.class` always starts with `"mvp-page"`

`get_page_class()` always prepends `"mvp-page"`:

```python
# page_class = "products-list"  →  page.class == "mvp-page products-list"
# page_class = ""               →  page.class == "mvp-page"
```

## MRO placement convention

Both mixins must appear **before** the Django generic view in the MRO:

```python
# Correct
class MyView(PageMixin, BaseTemplateNameMixin, DetailView): ...

# Incorrect — mixin methods won't intercept Django's method calls
class MyView(DetailView, PageMixin, BaseTemplateNameMixin): ...
```

## Testing your own subclass

```python
import pytest
from django.test import RequestFactory
from myapp.views import ProductDetailView
from myapp.models import Product

@pytest.fixture
def product(db):
    return Product.objects.create(name="Widget", ...)

def test_page_context(product, rf):
    request = rf.get("/products/1/")
    view = ProductDetailView()
    view.request = request
    view.kwargs = {"pk": product.pk}
    view.object = product

    context = view.get_context_data()
    assert context["page"]["title"] == "Widget"
    assert context["page"]["class"].startswith("mvp-page")
```
